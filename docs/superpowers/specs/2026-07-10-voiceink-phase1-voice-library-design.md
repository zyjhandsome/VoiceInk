## 1. 背景与总愿景

VoiceInk 当前(v1.3.5)是一款**本地离线、热键驱动、随处粘贴**的语音输入工具:热键 → 采集(麦克风/系统声/混合) → VAD 分段 → sherpa-onnx ASR → 可选 LLM 润色 → 校验粘贴。它最核心的资产是三点:**隐私/离线**、**极低使用摩擦**、**paste-anywhere 的通用性**。

下一阶段的产品形态定为:

> **VoiceInk 2.0 — 本地隐私的语音知识库。**
> 从「即时粘贴的输入工具」演进为「本地隐私的语音记录 + 可沉淀检索的知识库」。

形态三要素:

1. **paste-anywhere(口述,保持不变)** — 现有低摩擦主路径不动。
2. **会议主界面(新增,后续阶段)** — 面向多人会议/长录音的独立界面。
3. **统一本地语音库(贯穿全程)** — 无论口述还是会议,转写都自动沉淀进同一个本地库,可搜索、摘要、导出。

### 确认过的基础决策

| 决策点 | 结论 |
|--------|------|
| 库与 paste 的关系 | **passive 打底 → 演进到 dual**:先做被动副产品式的库(paste 主路径不变),再长出独立会议界面 |
| AI 与隐私边界 | **本地优先**:未来内置本地小模型做摘要/翻译,云 API 仅作高级可选项;前期先复用已有的可选 LLM 兜底 |
| 存储内容 | **仅文本**(不存音频),最轻、最隐私 |

## 2. 分期路线图

大愿景拆为 4 个可独立交付的子项目,每个各走一遍 spec → plan → 实现:

- **Phase 1 · 语音库地基(passive)** ← *本文档聚焦*
  每次转写自动存入本地库,新增「历史」窗口,可搜索/复制/删除/导出 Markdown。零破坏、纯增量、**完全无网络依赖**(按需摘要已移至 Phase 2,D3)。
- **Phase 2 · AI 沉淀**
  内置本地小模型做摘要/自动标签、库内语义搜索、导出到 Obsidian/Notion。
- **Phase 3 · 会议主界面(dual)**
  独立「录制/会议」窗口:系统声+麦混合长录音、实时滚动转写、(可选)说话人分离、结束一键纪要。
- **Phase 4 · 翻译/多模态**
  实时翻译、双语纪要。

## 3. Phase 1 详细设计:语音库地基(Passive Local Voice Library)

### 3.1 目标与边界

把「每次转写即时用完就丢」升级为「自动沉淀、可回看检索」。**零破坏**:现有热键 → 采集 → ASR → 润色 → 粘贴主路径完全不变,入库是在结果产出后**旁路**追加的一步,绝不阻塞或拖慢粘贴。

### 3.2 数据模型与存储

- 存储:`~/.voiceink/history.db`(SQLite,WAL 模式,`PRAGMA user_version=1`),文本-only。
- **粒度:一行 = 一个 segment**(方案 C,ADR-0001)。一场会话由同 `session_id` 的多行组成,聚合发生在查询层。
- 记录字段(每行):
  - `id` — 主键
  - `session_id` — 会话分组 id(持续模式一场共享;按住说话每次独立)
  - `seq` — 会话内段序号(保证拼接/导出顺序)
  - `created_at` — **该段定稿时刻**(会话时间 = 该 `session_id` 下 `MIN(created_at)`)
  - `raw_text` — 该段原始 ASR 文本(规范化后)
  - `polished_text` — 该段润色后文本(未开启润色则为空;降级为原文的段按 D5 也落 polished 侧)
  - `source` — 音频来源:`mic` / `system` / `mixed`
  - `duration_ms` — **该段**录音时长(`len(audio)/16000*1000`;会话总时长按需在查询层派生)
  - `target_app` — 粘贴目标前景**进程名**(D4,不存窗口标题;尽力而为,可空)
  - `trigger_mode` — `continuous` / `hotkey`
  - `model` — 识别所用模型名
  - ~~`summary`~~ — **Phase 1 冻结**(D3,推迟到 Phase 2)
- 搜索:**`LIKE '%q%'`(或 `instr`)全表扫描** `raw_text` + `polished_text`,**放弃 FTS5**(ADR-0002:FTS5 内置分词器搜不到中文双字词;≤5000 场会话量级 LIKE 毫秒级)。

### 3.3 写入时机与会话粒度(旁路,不阻塞输出)

- **逐段即时写行**(方案 C,ADR-0001):每个 segment 在 `_output_text` 产出后各写**一行**,不维护内存会话 buffer,不做「会话结束定稿写一条」。崩溃最多丢「最后一个尚未产出的段」,已产出的都在库里。
- **入队时机与口径**:写入入队挂在 `_handle_paste_result`(与 `target_app` 捕获同一刻,D4);记录口径 C —— `pasted`/`clipboard`/`error` 三分支只要文本非空一律入库(ADR-0004)。空文本不写。
- **异步写(ADR-0003)**:入队为「投递即忘」,后台写线程独占连接落库,永不进主/粘贴链路;退出时 flush。
- **会话粒度(查询层聚合)**:
  - 持续模式:一场 `session_id`(用户主动开始→用户 Esc/× 结束);**ASR 错误自动重启不得切分会话**(ADR-0009)。
  - 按住说话:每次松开 = 独立 `session_id`,`seq` 从 0。
- **记录组装(ADR-0010)**:每段记录逐槽渐进组装成不可变体并经粘贴回调闭包携带,入队时不回读 App 可变状态,规避持续模式段间抢跑污染。
- `HistoryStore` **不维护「当前会话」状态**;`session_id`/`seq` 由 `App` 按用户意图管理(ADR-0009)。

### 3.4 历史窗口(独立新窗口,托盘打开)

- 入口:托盘菜单新增「历史」项,打开独立窗口(为后续会议界面预留空间)。
- 列表:**按 `session_id` 分组**,时间倒序(会话时间取 `MIN(created_at)`),每条会话显示时间 / 来源 / 目标应用 / 首行摘要 / 段数;展开看各段(具体折叠交互留待 plan)。
- 搜索框:基于 **`LIKE` 实时过滤**(ADR-0002,可加简单去抖)。
- 单条操作:复制(原文 / 润色文本)、删除、导出 Markdown。**「按需生成摘要」已移除**(D3,推迟 Phase 2)。
- 批量操作:多选导出 Markdown(合并单文件,ADR-0008)、批量删除。
- 「清空全部历史」:独立显式按钮 + 二次确认,与总开关解耦(ADR-0007)。
- **写操作走单写者(ADR-0006)**:UI 删除/清空不直接执行 SQL,而是入队给后台写线程 → 回调刷新列表;读(列表/搜索/导出)用独立只读连接。
- 导出(ADR-0008):有效文本(有润色用 `polished_text` 否则 `raw_text`);单条一 `.md`、批量合并一 `.md`;YAML frontmatter(时间/来源/目标应用/时长/模型/段数);按 `seq` 拼接。
- 风格:复用现有 UI 样式/token(`voiceink/ui/`),保持 Apple 风观感。

### 3.5 隐私控制

- **历史记录总开关**:首启一次性询问(默认开,D1);**每次入队时读取**,运行中一关下一段即停;**只停未来写入、已有历史保留**(删除是独立动作,ADR-0007)。关闭后行为等同当前版本(用完即丢)。
- **记录口径**:非空文本一律入库(含粘贴降级/失败,ADR-0004);密码框不特殊处理,由总开关 + 删除兜底。
- **自动清理(ADR-0005)**:按「保留天数」或「最大**会话**数」上限,先到者触发,**整场删除最旧会话**(绝不半删一场),**活跃会话豁免**;计龄按会话 `MIN(created_at)`。默认 90 天 / 5000 场会话。启动时 + 每场会话结束后各跑一次,走后台写线程。
- **手动删除**:历史窗口内单条 / 批量删除 + 「清空全部」,均经单写者(ADR-0006)。

### 3.6 配置与设置

- 新增配置项并入 `~/.voiceink/config.json`(复用现有 `QTimer` 防抖保存机制):
  - `history.enabled`(bool,默认 true)
  - `history.onboarded`(bool,默认 false;首启询问后置 true,D1)
  - `history.retention_days`(int,默认 90)
  - `history.max_entries`(int,默认 5000 —— 计量单位为**会话数**,ADR-0005)
- 设置窗口新增对应开关(放置于哪一页在实现规划阶段确定)。

### 3.7 信号与架构约束

- 新增独立模块 `HistoryStore`(SQLite 封装,与 UI 解耦),由 `App` 在结果确定后调用。**单写者原则**(ADR-0006):对外只暴露两类 API —— 非阻塞的写/删/清理**入队**,与只读查询;所有写串行经后台写线程(ADR-0003),读用独立只读连接。
- **不新增、不改动** README 列出的四条核心信号(`ready`、`model_load_progress`、`segment_ready`、`esc_pressed`)。入库在既有处理槽(`_handle_paste_result` / `_output_text` / `_begin_transcription`)里**旁路调用**,保证 `App._connect_signals` 信号图不被破坏。
- **故障隔离(ADR-0011)**:`HistoryStore` 内部全程 try/except + 日志,不抛库异常回主线程;初始化失败则本次运行静默禁用历史,核心转写/粘贴照常。
- 遵守 README **变更审查清单**:同步更新 README,保证通过 `tests/test_readme_features.py`。
- 为 `HistoryStore` 新增单测:逐段写入原子性、`GROUP BY session_id` 聚合、**`LIKE` 子串搜索(含 1–2 字中文词)**、自动清理(按会话/不半删/活跃豁免)、总开关关闭时不入库、**会话不被 ASR 自动重启切分(ADR-0009)**、**段间抢跑不污染记录(ADR-0010)**、**写失败时主流程无感(ADR-0011)**、并发写入+UI 删除无 `database is locked`(ADR-0006)。

### 3.8 明确不做(YAGNI)

音频存储、本地小模型摘要、语义搜索、Obsidian/Notion 导出、会议界面、说话人分离、翻译——全部留给 Phase 2–4。

## 4. 风险与注意

- **不得拖慢粘贴**:入库必须异步/旁路,任何 DB 慢操作不能进入主输出链路(后台写线程,ADR-0003)。
- **信号图完整性**:这是 README 明令保护的红线,入库只能旁路挂接。
- **旁路不反噬主路径**:库故障(磁盘满/损坏/无权限/撞锁)一律吞掉记日志,绝不弄崩转写/粘贴(ADR-0011)。
- **会话完整性**:ASR 自动重启不得切分会话(ADR-0009);持续模式段间抢跑不得污染记录(ADR-0010)——二者均须有单测守护。
- **隐私默认值**:首启一次性询问(默认开,D1),设置中清晰可见且可一键关闭;总开关只停记录,清空历史为独立动作(ADR-0007)。
- **本地优先的代价**:Phase 2 内置本地小模型会显著增加打包体积(当前安装包已 ~800MB);Phase 1 **完全无网络依赖**(按需摘要已移除,D3)。
