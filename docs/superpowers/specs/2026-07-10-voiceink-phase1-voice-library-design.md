# VoiceInk 下一阶段产品形态 — 设计文档

> 日期:2026-07-10
> 状态:已与产品负责人确认方向，进入实现规划前的设计基线。

## 1. 背景与总愿景

VoiceInk 当前(v1.3.5)是一款**本地离线、热键驱动、随处粘贴**的语音输入工具:热键 → 采集(麦克风/系统声/混合) → VAD 分段 → sherpa-onnx ASR → 可选 LLM 润色 → 校验粘贴。它最核心的资产是三点:**隐私/离线**、**极低使用摩擦**、**paste-anywhere 的通用性**。

下一阶段的产品形态定为:

> **VoiceInk 2.0 — 本地隐私的语音知识库。**
> 从「即时粘贴的输入工具」演进为「本地隐私的语音记录 + 可沉淀检索的知识库」。

形态三要素:

1. **paste-anywhere(口述,保持不变)** — 现有低摩擦主路径不动。
2. **会议主界面(新增,后续阶段)** — 面向多人会议/长录音的独立界面。
3. **统一本地语音库(贯穿全程)** — 无论口述还是会议,转写都自动沉淀进同一个本地库,可搜索、摘要、导出。

### 确认过的基础决策

| 决策点 | 结论 |
|--------|------|
| 库与 paste 的关系 | **passive 打底 → 演进到 dual**:先做被动副产品式的库(paste 主路径不变),再长出独立会议界面 |
| AI 与隐私边界 | **本地优先**:未来内置本地小模型做摘要/翻译,云 API 仅作高级可选项;前期先复用已有的可选 LLM 兜底 |
| 存储内容 | **仅文本**(不存音频),最轻、最隐私 |

## 2. 分期路线图

大愿景拆为 4 个可独立交付的子项目,每个各走一遍 spec → plan → 实现:

- **Phase 1 · 语音库地基(passive)** ← *本文档聚焦*
  每次转写自动存入本地库,新增「历史」窗口,可搜索/复制/删除/导出 Markdown,复用现有可选 LLM 做按需单条摘要。零破坏、纯增量。
- **Phase 2 · AI 沉淀**
  内置本地小模型做摘要/自动标签、库内语义搜索、导出到 Obsidian/Notion。
- **Phase 3 · 会议主界面(dual)**
  独立「录制/会议」窗口:系统声+麦混合长录音、实时滚动转写、(可选)说话人分离、结束一键纪要。
- **Phase 4 · 翻译/多模态**
  实时翻译、双语纪要。

## 3. Phase 1 详细设计:语音库地基(Passive Local Voice Library)

### 3.1 目标与边界

把「每次转写即时用完就丢」升级为「自动沉淀、可回看检索」。**零破坏**:现有热键 → 采集 → ASR → 润色 → 粘贴主路径完全不变,入库是在结果产出后**旁路**追加的一步,绝不阻塞或拖慢粘贴。

### 3.2 数据模型与存储

- 存储:`~/.voiceink/history.db`(SQLite),文本-only。
- 记录字段:
  - `id` — 主键
  - `created_at` — 创建时间(会话结束定稿时间)
  - `raw_text` — 原始 ASR 文本(会话内多 segment 拼接后的完整文本)
  - `polished_text` — 润色后文本(未开启润色则为空)
  - `source` — 音频来源:`mic` / `system` / `mixed`
  - `duration_ms` — 录音/会话时长
  - `target_app` — 粘贴目标前景窗口进程名(尽力而为,可空)
  - `trigger_mode` — `continuous` / `hotkey`
  - `model` — 识别所用模型名
  - `summary` — 按需生成的摘要(默认空)
- 全文搜索:SQLite **FTS5**,索引建在 `raw_text` + `polished_text` 上。

### 3.3 写入时机与会话粒度(旁路,不阻塞输出)

- 结果最终确定后写库:润色流程在 `polish_complete` / 降级路径后写;纯 ASR 在 `final_result` 后写。写库在**粘贴之后**异步进行。
- **持续模式按会话聚合**:一次持续监听从开始到 Esc/结束聚合为**一条**记录,期间每个 segment 追加拼接到该会话的文本;会话结束时定稿。
- **按住说话模式**:每次松开热键定稿为**一条**记录。
- `HistoryStore` 需维护「当前会话」状态:持续监听开始建会话 → 逐 segment 追加 → 结束定稿。

### 3.4 历史窗口(独立新窗口,托盘打开)

- 入口:托盘菜单新增「历史」项,打开独立窗口(为后续会议界面预留空间)。
- 列表:时间倒序,每行显示时间 / 来源 / 目标应用 / 文本首行摘要。
- 搜索框:基于 FTS5 实时过滤。
- 单条操作:复制(原文 / 润色文本)、删除、导出 Markdown、**按需生成摘要**(复用现有可选 LLM;未配置 API 时该按钮禁用并提示)。
- 批量操作:多选导出 Markdown、批量删除。
- 风格:复用现有 UI 样式/token(`voiceink/ui/`),保持 Apple 风观感。

### 3.5 隐私控制

- **历史记录总开关**(默认开):可一键彻底关闭入库;关闭后行为等同当前版本(转写用完即丢)。
- **自动清理**:按「保留天数」或「最大条数」上限,超出自动删除。默认建议 90 天 / 5000 条(取先到者)。
- **手动删除**:历史窗口内单条 / 批量删除。

### 3.6 配置与设置

- 新增配置项并入 `~/.voiceink/config.json`(复用现有 `QTimer` 防抖保存机制):
  - `history.enabled`(bool,默认 true)
  - `history.retention_days`(int,默认 90)
  - `history.max_entries`(int,默认 5000)
- 设置窗口新增对应开关(放置于哪一页在实现规划阶段确定)。

### 3.7 信号与架构约束

- 新增独立模块 `HistoryStore`(SQLite 封装,与 UI 解耦),由 `App` 在结果确定后调用。
- **不新增、不改动** README 列出的四条核心信号(`ready`、`model_load_progress`、`segment_ready`、`esc_pressed`)。入库在既有处理槽(`_on_final_result` / 润色完成后续)里旁路调用,保证 `App._connect_signals` 信号图不被破坏。
- 遵守 README **变更审查清单**:同步更新 README,保证通过 `tests/test_readme_features.py`。
- 为 `HistoryStore` 新增单测:写入、会话聚合、FTS 搜索、自动清理、总开关关闭时不入库。

### 3.8 明确不做(YAGNI)

音频存储、本地小模型摘要、语义搜索、Obsidian/Notion 导出、会议界面、说话人分离、翻译——全部留给 Phase 2–4。

## 4. 风险与注意

- **不得拖慢粘贴**:入库必须异步/旁路,任何 DB 慢操作不能进入主输出链路。
- **信号图完整性**:这是 README 明令保护的红线,入库只能旁路挂接。
- **隐私默认值**:总开关默认开、但需在设置中清晰可见且可一键关闭,符合本地隐私心智。
- **本地优先的代价**:Phase 2 内置本地小模型会显著增加打包体积(当前安装包已 ~800MB),故摘要在 Phase 1 仅复用可选云 LLM。
