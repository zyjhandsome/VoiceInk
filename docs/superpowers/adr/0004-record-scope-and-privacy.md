# ADR-0004 · 入库记录口径与隐私边界

> 日期:2026-07-10 · 状态:已接受 · 关联:spec §3.3/§3.5、决策补丁 D1/D4
> 来源:grill-with-docs 拷问 Question 4(哪些事件产出记录)

## 背景

`_handle_paste_result` 有三个结果分支:`pasted`(确认粘贴)、`clipboard`(文本已复制、
粘贴未确认,如管理员窗口/密码框/沙箱)、`error`(连剪贴板都失败)。三者中文本均已成功
产出(转写/润色完成)。需定义哪些分支写入历史库。空文本已在 `_output_text` 前置 return,
不在讨论范围。

## 决策

**记录口径 = C:只要产出非空文本就入库,含全部三个分支(`pasted` / `clipboard` / `error`)。**

- 写入入队挂在 `_handle_paste_result`(与 D4 的 `target_app` 捕获同一时刻)。
- **密码框不做特殊处理**:VoiceInk 无法可靠识别密码框,特殊处理是虚假安全感;
  由「历史总开关(D1)+ 手动删除」兜底,符合本地库心智。
- **正面性质**:`error` 分支下用户未拿到文本,历史库成为找回该段的唯一兜底。

**否决**
- A(仅 `pasted`):降级到剪贴板的常见场景(管理员/密码框/沙箱/自窗口保护)会丢档,
  与「知识库」定位冲突。
- B(仅 `pasted`+`clipboard`):比 A 好,但仍丢掉 `error` 这一「最该被找回」的场景。
  (若后续要收紧,B 是唯一合理的退路。)

## 影响

- `HistoryStore` 写入触发点统一在 `_handle_paste_result` 三分支末尾入队。
- 需注意:同一 segment 只入队一次(避免 pasted 后又被其它路径重复写)。
- 测试:三个分支各产出一行;空文本不产出;总开关关闭时任何分支都不写。
