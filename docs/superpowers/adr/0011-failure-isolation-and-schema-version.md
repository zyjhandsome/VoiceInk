# ADR-0011 · 历史库故障隔离与 schema 版本

> 日期:2026-07-10 · 状态:已接受 · 关联:ADR-0003、ADR-0006、风险「零破坏 / 旁路不反噬」
> 来源:grill-with-docs 拷问 Question 11(故障隔离)

## 背景

历史库会遇到现实故障:磁盘满、`history.db` 损坏、WAL 落盘失败、`~/.voiceink/` 无写权限、极端撞锁。
这些绝不能冒泡到转写/粘贴主链路,否则「旁路副产品」把「随处粘贴」核心功能弄崩,违背 Phase 1 头号承诺。

## 决策

1. **故障隔离不变量**:后台写线程内**任何**异常(建库/建表/INSERT/DELETE/清理)一律**吞掉并记日志**,
   绝不抛回主线程、绝不进出字链路。`enqueue` 是「投递即忘」——永不阻塞、永不因写失败报错给调用方。
2. **初始化失败降级**:若启动时库都打不开(权限/损坏),**本次运行静默禁用历史**
   (运行时等效 `history.enabled=false`),核心转写/粘贴照常;在设置/日志给一次可见提示,不弹错、不拦路。
3. **schema 版本**:建库写 `PRAGMA user_version`(Phase 1 = 1),为 Phase 2 加字段
   (summary/tags/embeddings)预留前向迁移钩子。Phase 1 只建表 + 记版本,不写迁移逻辑。

## 影响

- `HistoryStore` 内部全程 try/except + 日志;对外 API 不抛库相关异常。
- `App` 侧:历史相关调用不放入任何影响输出/粘贴的判断路径。
- 读路径(历史窗口)同样容错:查询失败显示空列表 + 提示,不崩窗口。
- 测试(红线):注入写失败/初始化失败 → 转写/粘贴主流程完全无感;`enqueue` 不抛;初始化失败时运行时降级。
