# ADR-0006 · 单写者原则(所有写操作串行经后台写线程)

> 日期:2026-07-10 · 状态:已接受 · 关联:ADR-0003、ADR-0005、spec §3.4/§3.7
> 来源:grill-with-docs 拷问 Question 6(UI 删除与后台写的并发)

## 背景

历史窗口提供单条删除、批量删除、导出。删除是写操作。若 UI 在主线程用**另一个连接**
`DELETE`,而后台写线程同时 `INSERT`(边看历史边持续口述),会形成**两个写者**,
WAL 下写-写撞锁 → 偶发 `database is locked`,难复现难查。

## 决策

**确立单写者原则:所有写操作(流水线 INSERT、自动清理 DELETE、UI 手动/批量删除)
一律通过唯一的后台写线程串行执行;只有读(列表、搜索、导出 SELECT)用独立只读连接。**

- UI 删除 → `enqueue` 删除操作给写线程 → 完成回调刷新列表(不直接执行 SQL)。
- `HistoryStore` 对外只暴露两类 API:
  1. **非阻塞写/删/清理入队**(pipeline 写入、手动删除、批量删除、自动清理)。
  2. **只读查询**(列表分组、LIKE 搜索、导出读取)。
- 从根上消除写-写竞争,写顺序确定可测。

## 影响

- UI 删除由「同步」变为「入队 + 回调刷新」,借 Qt 信号/`singleShot` 接线,与现有风格一致。
- 只读连接与写连接分离;数据库用 WAL 模式允许读写并发(读不阻塞写)。
- 测试:并发「持续写入 + UI 删除」无 `database is locked`;删除经写线程后列表最终一致。
