# ADR-0002 · 搜索用 LIKE 全表扫描,放弃 FTS5

> 日期:2026-07-10 · 状态:已接受 · 关联:spec §3.2/§3.4/§3.7、决策补丁 D6
> 来源:grill-with-docs 拷问 Question 2(搜索方案),含真实环境实测

## 背景

spec §3.2/§3.7 把「SQLite FTS5」定为全文搜索地基。但 VoiceInk 中文为主,
且中文句子通常无词间空格。在实际打包所用的 Python(sqlite 3.50.4)上实测:

句子:`今天天气很好我们去公园散步`(无空格)

| 方案 | `天气`(2字) | `公园散步`(4字) | `散步`(2字) |
|------|:---:|:---:|:---:|
| FTS5 `unicode61`(默认) | 0 | 0 | 0 |
| FTS5 `trigram` | 0 | 1 | 0 |
| `LIKE '%q%'` 全表扫描 | 1 | 1 | 1 |

- `unicode61` 把整句当一个 token,子串全灭。
- `trigram` 仅匹配 ≥3 字查询,最常见的双字词(天气/散步/会议/报告)搜不到。
- `LIKE` 对 1–2 字中文子串全部命中。

## 决策

**Phase 1 搜索改用 `LIKE '%q%'`(或 `instr`)全表扫描,放弃 FTS5。**

理由:
1. **中文子串检索正确性**:LIKE 覆盖 1–2 字词,FTS5 内置分词器做不到。
2. **规模匹配**:库上限 ≤5000 条短文本(决策补丁按 session 计数,行数同量级),
   LIKE 全扫亚毫秒~毫秒级,无需倒排索引。
3. **复杂度更低**:免去 FTS5 外部内容表 / 触发器同步、tokenizer 选择、
   与 raw/polished 双字段的一致性维护。
4. **零网络/轻量约束**:排除需要 jieba/ICU 外部分词器的方案(打包已 ~800MB)。

**否决的备选**
- FTS5 + trigram:双字词搜不到,或需再叠一层 LIKE 回退 = 两套都维护。
- FTS5 + 外部中文分词器:依赖/体积与项目约束冲突。

## 影响

- spec §3.2 删除「全文搜索:FTS5,索引建在 raw_text+polished_text」→ 改为 LIKE 扫描 raw_text/polished_text。
- spec §3.4 搜索框仍「实时过滤」,底层实现改为 LIKE 查询(可加简单去抖)。
- spec §3.7 `HistoryStore` 单测:删除「FTS 搜索」项,改为「LIKE 子串搜索(含 1–2 字中文词)」。
- 术语表:移除「FTS5 索引字段」措辞。
- **未来**:若 Phase 2 引入语义搜索,是另起方案(向量),与本决策不冲突。
