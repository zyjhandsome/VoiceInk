# ADR-0003 · 写库走专用后台线程 + 队列

> 日期:2026-07-10 · 状态:已接受 · 关联:spec §3.3/§3.7、风险「不得拖慢粘贴」
> 来源:grill-with-docs 拷问 Question 3(写库执行模型)

## 背景

`text_paster.paste_async` 用 `QTimer.singleShot` 派发,`_handle_paste_result` 及其触发的
写库调用都跑在 **Qt 主线程**。粘贴按键在回调前已模拟完成,写库发生在粘贴之后。
需确定:写库同步在主线程做,还是丢到后台线程。

## 决策

**采用后台写线程 + 队列(方案 B)。**

- `HistoryStore` 内部持有**单个后台写线程**,该线程**独占一个 SQLite 连接**。
- 出字路径(`_output_text` 成功后)只把一条待写记录 `put` 进线程安全队列,**非阻塞**,立即返回。
- 后台线程消费队列做 INSERT(WAL 模式)。
- 应用退出时 **flush 队列**并优雅关闭连接(等待队列排空或有界超时)。

理由:
1. **红线结构化**:「写库永不进入主/粘贴热路径」成为架构事实,而非「大概够快」。
2. **契合 spec §3.3 原话「异步」**。
3. **`check_same_thread` 天然解决**:连接只属于写线程,不跨线程共享。
4. **Phase 3 复用**:会议模式短时间大量分段,同一异步通道可直接承接。

**否决**:A(主线程同步写)——虽字面满足「不拖慢粘贴按键」,但极端磁盘卡顿会 jank UI,
且不匹配 spec「异步」措辞;仅在极简优先时可退回。

## 影响

- `HistoryStore` 接口:提供非阻塞的 `enqueue(record)`(或 `record_segment(...)`)+ `close()/flush()`。
- 读路径(历史窗口查询/搜索)与写线程分离:读可用独立连接(WAL 允许并发读)。
- 需在 `App` 退出流程挂接 `HistoryStore.close()` 以保证 flush。
- 测试:入队非阻塞、后台落库最终一致、退出 flush 不丢队列、写线程连接单一。
